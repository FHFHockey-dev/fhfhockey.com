### Formula, window, and hidden-assumption map

- `playerTrendCalculator` sorts by player/date, accumulates finite non-null values, publishes cumulative mean plus sample variance/standard deviation, and publishes rolling 3/5/10 only after a complete window. State resets by player; the scheduled writer supplies an exact current season, while the public optional-season POST can span seasons. Skater metrics use canonical view fields with selected fallbacks; goalie metrics contain multiple `any` casts. `expected_shooting_pct` is not semantically compatible with the contextual ranking definition (P1 NEW 2).
- `teamPercentiles` ranks each metric by ordinal index `(index / (n-1)) * 100`, reverses lower-is-better metrics, assigns a one-team population 50, then takes the configured weighted mean of available metric percentiles. Missing metrics reduce the denominator rather than fail the category. Ties are not midpoint-ranked, so equal values can receive different percentiles according to stable input order.
- `teamMetricConfig` defines offense, defense, power-play, and penalty-kill bundles with fixed integer weights 1–5 across AS/PP/PK/WGO sources. Opportunity counts participate as positive offense and negative defense context; no normalization by games occurs in the percentile helper beyond whatever each source column already represents.
- `skaterMetricConfig` exposes shots/60, ixG/60, TOI, and PP TOI; all are marked higher-is-better. Allowed windows are 1/3/5/10/20, default 1; position cohorts are forwards, defense, or all; limit is 25 by default and 50 maximum.
- `lib/trends/utils` uses population standard deviation, a first-greater-or-equal percentile lookup, 1% default winsorization, ordinary least squares, fixed luck baselines (8% on-ice shooting and PDO 1.0), fixed SKO weights, and partial-window rolling averages. Sandbox hot/cold streaks compare raw points—not rolling values—to one caller-provided baseline and require two consecutive games; intensity is `min(1, 0.25 + 0.18 * length)`.

### Formula and data-ownership map

- `priors.ts` pools F/D league means for shooting percentage, on-ice shooting percentage, individual-points percentage, and PP shooting percentage, converts each mean and configurable strength `k` into Beta parameters, and constructs three-season player posteriors with explicit rookie fallback thresholds.
- `windows.ts` aggregates the most recent 3/5/10/20 games as successes/trials, blends sample and prior variance through `shrink = 1 / (1 + n/k)`, and persists rate, prior, shrinkage, mixed variance, and empirical-Bayes z-score rows.
- `score.ts` clips skill z-scores to ±3, applies fixed negative luck weights and positive ixG/iCF/HDCF skill weights, maps the result through guarded 0–100 scoring, and embeds model/config/source-cutoff/warning provenance in JSON components. First-class version columns remain the separate B-SUST-BAR NEW 15 boundary.
- `bandCalculator.ts` uses metric-specific Gamma, Beta, or Normal families, half-life weighting, prior blending, and credible intervals for l3/l5/l10/l20 outputs. `bandService.ts` owns source loading, calculation, and composite-key persistence.
- `lib/baselines/*` is the older configurable snapshot-window aggregation layer. It writes `player_baselines`; it is an input to the canonical TypeScript window-z chain, not an alternate score writer.

## Team power / underlying / matchup static audit (`3.1`–`3.4`, `3.6`–`3.7`)

- `/underlying-stats` is the canonical persisted-rating diagnosis page. It reads `/api/underlying-stats/team-ratings`, displays offense, defense, pace, power score, trend10, PP/PK tiers, SOS, components, and responsive table/card views. `/trends` uses the separately computed trend-category/CTPI/SOS endpoints for movement context; `trends/placeholder` duplicates those concepts as a legacy alternate.
- `update-team-power-ratings` is the canonical `team_power_ratings_daily` writer. It range-paginates NST/WGO inputs, uses a 25-game exponentially weighted window with ten-game half-life, shrinks the first ten games toward league means, computes league z-scores, and normalizes offense/defense/pace to mean 100 and SD 15. The `-new` route is a no-write quarantine stub for `team_power_ratings_daily__new`, so the two writers are not co-authoritative.
- `teamRatingsService` reads the canonical table with a short cache, gracefully falls back when extended columns are absent, and falls back from an empty requested date to the latest available snapshot. Team-power trend categories are a distinct in-memory percentile series over AS/PP/PK/WGO inputs, not the persisted 100±15 rating model.
- CTPI prefers `team_ctpi_daily`, then recomputes from paginated AS/PP/PK inputs. Its composite is 35% offense, 30% defense, 20% goaltending, 15% special teams, plus a 10% PDO/luck term. SOS reads the latest `sos_standings` snapshot and reports past/future/combined opponent win percentages and rankings.
- Approved/current formulas are the canonical persisted 25-game EWMA ratings, current in-memory category percentile configuration, CTPI weighting, and standings-backed SOS. The quarantined `__new` path is legacy. `power-ratings.ts` explicitly questions whether missing PDO is 1.0 or 100 and currently substitutes 100; this is suspect and retained as P1 NEW 6 rather than silently changing historical ratings.

## FORGE pipeline static audit (`5.1`–`5.5`)

- `/FORGE` presents run-scoped skater, team, and goalie projections, confidence/uncertainty ranges and drivers, calibration/health context, requested/resolved-date serving state, and navigation into player trends and Start Chart. Its unique ownership is projection output and uncertainty; rolling metrics, team power, and player trends remain inputs or linked context rather than duplicate writers.
- The `/api/v1/projections/{players,teams,goalies}` compatibility readers use bounded query schemas, optional date/horizon/run filters, and latest-succeeded-run resolution over `forge_player_projections`, `forge_team_projections`, and `forge_goalie_projections`. - The authoritative pipeline has eight ordered stages: roster/schedule; line combinations; projection-input ingest; shift-chart relationship construction; player/team/goalie derived strength; v2 goalie-start priors; the v2 FORGE projection run; and accuracy/calibration. The stage contract records every dependency and produced object.
- `update-goalie-projections-v2` is the only supported `goalie_start_projections` writer. The old goalie writer was first quarantined and then deleted after a clean in-repository caller audit. `run-projection-accuracy` compares projection rows with realized games/WGO/goalie outcomes and persists run, daily, player, stat, and calibration diagnostics.

## Consolidation decision register

The target is one shared data/model layer with multiple purpose-specific views, not one formula that erases distinct semantics.

- **Keep as authoritative engines:** canonical rolling-player builder after NEW 10 repair; canonical Sustainability priors/window-z/score/bands after NEW 8 and B-SUST-BAR provenance decisions; canonical team ratings after NEW 6/9; FORGE run/projection/uncertainty tables after remaining NEW 13 coverage repair. NEW 12's reader adapter is closed locally.
- **Keep as presentation views:** `/trends` for movement/context, Start Chart for slate decisions, player detail for raw rolling diagnosis, and FORGE for forecast confidence. A future dashboard can expose these as tabs over shared typed readers and a common requested/resolved-date/provenance contract.
- **Refactor/merge adapters:** centralize fantasy scoring, requested/observed source cutoffs, team identity, and confidence-driver extraction. Reuse the canonical run resolver and date/fallback metadata rather than duplicating projection math.
- **Deprecate after caller proof:** the Trends placeholder, Testing Grounds once operator diagnostics move, compatibility projection readers after all consumers migrate, and WIGO recent/career surfaces after explicit caller/data-retention review. The deleted legacy Start Chart writer stays retired.
- **Required regression gates:** mixed regular/playoff units and accumulator history; requested-versus-observed freshness; late-arriving team source repair; full-history date-scoped rolling reconstruction and support payload parity; shared fantasy scoring; nested FORGE confidence metadata; player/team/goalie arithmetic; accuracy eligibility/coverage; and API-to-visible-UI bindings.
- **Execution boundary:** no consolidation deletion, historical rewrite, or model promotion is authorized by this audit. NEW 2/6–10/13 and B-SUST-BAR NEW 13–15 remain explicit prerequisites, each with bounded repair/backfill and version evidence; NEW 11's versioned read-time adapter is complete.
