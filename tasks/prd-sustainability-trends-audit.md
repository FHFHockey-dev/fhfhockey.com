# Sustainability / Trends Audit PRD

## 1. Introduction / Overview

This PRD defines a comprehensive audit of all pages, APIs, helper modules, and background jobs that estimate, summarize, or compare a player's or team's typical performance using historical data. This includes trend, sustainability, baseline, rolling average, projection, and power-rating systems.

The immediate goal is not to rewrite the code. The immediate goal is to produce a disciplined audit that explains:

- What each system is trying to measure
- What raw and derived data each system relies on
- How each system calculates its outputs
- Which Supabase tables each page and API reads from or writes to
- Which systems overlap, conflict, drift in definition, or appear stale
- Which pieces should survive into one future unified engine and one future visual dashboard

The end state after the audit is expected to be one consolidated calculation engine with one primary visual dashboard page that can expose multiple tabs or views, instead of multiple semi-overlapping pages and experiments.

## 2. Goals

- Produce a complete inventory of repo surfaces that calculate or expose trends, sustainability, projections, baselines, rolling averages, or historical-performance-derived outputs.
- Audit one page or folder at a time so the work is traceable and does not become another sprawling cleanup effort.
- For every audited page, document direct helpers, imported logic, related API routes, background builders, and Supabase dependencies.
- For every relevant API endpoint and job, document source tables, derived tables, and explicit upsert targets.
- Define repeatable validation procedures that compare raw player/team data against the outputs generated by each calculation path.
- Identify duplication, definition drift, stale prototypes, and conflicting metrics across the repo.
- Produce a decision-ready consolidation map for a future unified engine and visual dashboard.
- Preserve useful ideas from experiments instead of deleting them blindly.

## 3. User Stories

- As the maintainer, I want a complete map of all trend/sustainability/baseline systems so I can stop guessing which file does what.
- As the maintainer, I want to validate calculations against raw player and team data so I can trust the logic before consolidating anything.
- As the maintainer, I want to understand which tables are raw, derived, cached, or authoritative so I can reduce data confusion.
- As the maintainer, I want old and experimental files included in the audit so hidden duplicate logic does not survive by accident.
- As the maintainer, I want a visual end goal after the audit so I can inspect results in a UI instead of relying only on SQL rows or JSON payloads.
- As a future developer, I want a junior-readable audit plan so I can execute the work step by step without reverse-engineering the whole repo first.

## 4. Functional Requirements

1. The audit PRD must define the audit as a page-by-page and folder-by-folder process, not as one broad undifferentiated review.
2. The audit must include the user-listed pages:
   - `web/pages/trendsSandbox.tsx`
   - `web/pages/trendsTestingGrounds.tsx`
   - `web/pages/trends/index.tsx`
   - `web/pages/FORGE.tsx`
   - `web/pages/underlying-stats/index.tsx`
   - `web/pages/trends/placeholder.tsx`
   - `web/pages/start-chart.tsx`
3. The audit must also include additional repo surfaces discovered by scan if they compute similar logic, including current, legacy, experimental, or partially abandoned implementations.
4. The audit must include both user-facing pages and the background builders/upsert jobs that feed them.
5. The audit must separate findings into system families at minimum:
   - Trends / skater trend metrics
   - Sustainability / baselines / bands / scores
   - Team power / underlying stats / matchup context
   - Start-chart / rolling projection logic
   - FORGE / projection engine
   - Legacy or adjacent average/baseline systems
6. For each audited page, the audit must record:
   - Purpose of the page
   - User-facing outputs and metric names
   - Imported helpers and modules
   - Related API endpoints
   - Direct Supabase reads
   - Indirect Supabase dependencies through APIs or helper modules
   - Known sibling or duplicate pages
7. For each audited API endpoint, the audit must record:
   - Route path
   - HTTP methods
   - Input parameters
   - Tables/views/functions read from
   - Tables/views written or upserted to
   - Whether the endpoint is read-only, recompute-only, or both
   - Which pages depend on it
8. For each audited background job or builder, the audit must record:
   - Trigger path or script entry point
   - Calculation responsibility
   - Source tables/views/functions
   - Derived or target tables
   - Whether it is authoritative, supplemental, or legacy
9. The audit must explicitly document table ownership and role classification for each relevant table:
   - Raw source
   - Derived aggregate
   - Cached output
   - Presentation-ready output
   - Historical snapshot
   - Unknown / mixed responsibility
10. The audit must explicitly capture which endpoints upsert into which tables, including at minimum the currently observed surfaces:
   - `player_trend_metrics`
   - `sustainability_scores`
   - `sustainability_trend_bands`
   - `player_baselines`
   - `team_power_ratings_daily`
   - `team_power_ratings_daily__new`
   - `rolling_player_game_metrics`
   - `player_projections`
   - `goalie_start_projections`
   - `forge_player_projections`
   - `forge_team_projections`
   - `forge_goalie_projections`
   - `wigo_recent`
   - `wigo_career`
11. The audit must require a terminology and definition sheet for each system that answers:
   - What “trend” means in that file
   - What “baseline” means in that file
   - What “projection” means in that file
   - Whether the output is descriptive, predictive, comparative, or blended
   - What sample window is used
   - Whether the logic is player-level, team-level, goalie-level, or matchup-adjusted
12. The audit must require an overlap matrix that shows where different pages or systems calculate similar outputs with different formulas, windows, or source tables.
13. The audit must define a canonical validation dataset covering both player and team cases, including:
   - High-sample skater
   - Low-sample skater
   - Defenseman
   - Recently moved or irregular-usage player if available
   - One goalie if goalie trend/projection logic is involved
   - One stable team
   - One volatile team
14. The audit must require a single-player trace and single-team trace for every major system:
   - Pull raw source rows
   - Show intermediate derived values
   - Run the calculation path
   - Compare final output to hand-checked expectations
15. The audit must define validation steps for both direct computation and persistence:
   - Verify raw input rows
   - Verify intermediate transformations where available
   - Verify final API response or page output
   - Verify target table rows after rebuild jobs run
16. The audit must define SQL/API verification scripts or repeatable command snippets for the audit, not just manual eyeballing.
17. The audit must define a final regression test suite to be built after the audit so the future consolidated system can be compared against approved logic.
18. The audit must classify each file or subsystem into a future-state recommendation:
   - Keep
   - Refactor into shared helper
   - Merge into unified engine
   - Keep as UI-only wrapper
   - Deprecate
   - Delete
19. The audit must end with a consolidation decision framework that supports building one future visual dashboard with multiple tabs/views powered by one underlying engine.

## 6. Audit Execution Plan

### Phase 1: Inventory and Grouping

- Build the master inventory of in-scope pages, APIs, helpers, jobs, SQL functions, and related tests.
- Group all items into system families.
- Mark each item as current production, active experiment, legacy, or unclear.
- Produce a dependency map from page to API to helper to table.

### Phase 2: Page-by-Page Audit

- Audit one page or page-family at a time.
- For each page, document all direct imports, related API endpoints, and output metrics.
- Record every visible metric name and what it appears to mean.
- Record UI duplication or conceptual overlap with other pages.

Suggested page order:

1. `web/pages/trends/index.tsx`
2. `web/pages/trends/placeholder.tsx`
3. `web/pages/trendsSandbox.tsx`
4. `web/pages/trendsTestingGrounds.tsx`
5. `web/pages/underlying-stats/index.tsx`
6. `web/pages/start-chart.tsx`
7. `web/pages/FORGE.tsx`
8. `web/pages/trends/player/[playerId].tsx`
9. Adjacent legacy pages with overlapping rolling-average logic

### Phase 3: API and Builder Audit

- For each page-family, audit the APIs and jobs that feed it before moving to the next family.
- Document source and target tables.
- Document if any route both computes and serves data.
- Document freshness assumptions, snapshots, and fallback behavior.

Suggested family order:

1. Player trend metrics
2. Sustainability scores / bands / baselines
3. Team power / underlying ratings / SOS / CTPI
4. Start-chart projections and rolling-player inputs
5. FORGE projections and accuracy reporting
6. Legacy average and WIGO-style systems

### Phase 4: Validation and Hand-Checks

- Run raw-data pulls for canonical players and teams.
- Re-run relevant rebuild jobs for narrow test scopes where possible.
- Compare raw inputs, intermediate values, persisted rows, API responses, and UI outputs.
- Log mismatches, unexplained deltas, naming collisions, and hidden assumptions.

### Phase 5: Consolidation Decisioning

- Produce overlap matrix and quality ranking.
- Mark preferred formulas and preferred data sources.
- Identify which helpers can become shared engine logic.
- Identify which pages can collapse into one future dashboard with separate tabs.

## 7. Validation / Test Plan

### 7.1 Core Validation Pattern

For each major system, the audit must run this pattern:

1. Pick one canonical entity for the test.
2. Pull the raw rows from the underlying source tables or views.
3. Confirm the sample window, filters, and ordering used by the calculation.
4. Run the endpoint or rebuild path that calculates the output.
5. Pull the resulting target rows from Supabase.
6. Compare raw values, intermediate calculations, and final outputs.
7. Record whether the result is correct, ambiguous, inconsistent, or unverifiable.

### 7.2 Required Validation Modes

- SQL verification of raw source rows
- API verification of read endpoints
- Rebuild verification of write/upsert endpoints
- UI verification that displayed values match backend results
- Regression verification after preferred formulas are identified

### 7.3 Canonical Test Cases

- Skater A: high-sample forward
- Skater B: lower-sample or volatile forward
- Skater C: defenseman
- Player D: edge case such as recent move, role shift, or missing-game pattern
- Goalie E: for starter probability, trend, or FORGE-related logic
- Team A: stable strong team
- Team B: volatile or average team

### 7.4 Required Data Checks

- Date window correctness
- Season filter correctness
- Position filter correctness
- Per-game vs per-60 consistency
- Handling of missing/null rows
- Handling of small sample sizes
- Handling of stale snapshots
- Handling of fallback dates and fallback tables
- Consistency between page naming and formula meaning

### 7.5 Post-Audit Regression Suite Requirements

- Add endpoint-level tests for approved trend and sustainability contracts.
- Add fixture-driven tests for selected player and team calculations.
- Add persistence tests for upsert jobs writing to the approved target tables.
- Add one comparison suite that verifies the future unified engine against approved legacy benchmark cases during migration.

## 8. Design Considerations

- The audit itself is documentation-first, but it must explicitly support a later visual dashboard.
- The future end state should be one primary page with multiple tabs or views rather than many separate near-duplicate pages.
- The future UI must expose calculation context clearly enough for a non-database-first workflow:
  - raw inputs
  - selected windows
  - intermediate values where useful
  - final outputs
  - source table / endpoint provenance
- The audit should identify which current pages already offer useful visual patterns worth reusing in that final dashboard.

## 9. Technical Considerations

- Several systems use overlapping but different source layers:
  - raw tables
  - unified views
  - rolling aggregate tables
  - snapshot tables
  - projection output tables
- Some endpoints are read-only while others both compute and persist data.
- Some systems include fallback table behavior, cached responses, or date fallback behavior that must be documented as part of correctness.
- Some systems may be authoritative only for UI and not for data generation.
- Some systems may rely on stale or duplicate derived tables that should not survive consolidation.
- Audit deliverables should distinguish:
  - source of truth
  - current UI dependency
  - likely replacement candidate

## 10. Non-Goals (Out of Scope)

- This PRD does not implement the audit yet.
- This PRD does not redesign the final dashboard UI yet.
- This PRD does not consolidate or delete files yet.
- This PRD does not optimize performance unless a performance issue affects correctness or reproducibility.
- This PRD does not replace the current systems during discovery.
- This PRD does not define the exact final formulas for the unified engine; it defines how those formulas will be selected through audit evidence.

## 11. Success Metrics

- 100% of in-scope pages and discovered sibling systems are inventoried.
- Every audited page has a documented dependency chain from page to API to helper to Supabase source/target tables.
- Every relevant calculation endpoint or job has explicit read/write table documentation.
- Every major system has at least one player or team trace completed end-to-end.
- Every major overlap or duplicate calculation path is categorized as keep, merge, refactor, deprecate, or delete.
- The audit produces a final shortlist of approved source tables, approved formulas, and approved UI concepts for the future unified dashboard.
- The audit materially reduces ambiguity about which file should be trusted for a given metric.

- Metric definition glossary
