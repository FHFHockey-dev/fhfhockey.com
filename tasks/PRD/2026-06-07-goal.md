# 10. Acceptance criteria

This goal is complete only when:

1. `/rankings` defaults to a multi-metric player matrix, not a single-metric-only leaderboard.
2. The existing single-metric contextual leaderboard remains available as `Metric Explorer`.
3. The matrix endpoint returns grouped metric cells with raw value, formatted value, rank, percentile, directionality, availability, sample confidence, methodology, and caveat metadata.
4. The UI renders metric-by-metric color-coded percentile/rank cells across player rows.
5. Sorting by visible metric columns works and updates URL state.
6. Filters work for player type, deployment, window, season, minimum GP, strength, and team.
7. Planned metrics are visibly planned/unavailable rather than shown as zero or silently omitted.
8. True `5v5` requests use true verified `5v5` data paths, not `ev` aliases.
9. The right-side player snapshot updates when a row is selected, using live row data for available metrics and planned/unavailable states for composite/archetype/regression metrics that are not yet implemented.
10. Loading, empty, unavailable, stale-source, low-sample, and small-peer-group states are clear.
11. Desktop layout matches the intended heatmap-workstation direction.
12. Mobile/tablet layouts do not overlap or break.
13. Lint and build pass.
14. Focused tests for the new API, registry, table, tabs, and URL state pass.
15. Any remaining blockers are appended as new tasks instead of buried in notes.
16. Tasks 53.0 and 54.0 are not fully implemented unless already-existing verified code/data made them trivial to surface.
17. Composite, archetype, regression, and WAR-style features are visibly planned/unavailable where not implemented.
18. No fake composite values, temporary client-side composite approximations, or unverified formulas are shown.
19. Any remaining work for 53.0 and 54.0 is recorded as follow-up task notes instead of being partially hidden in the implementation.

Stop and ask for direction only if:

- implementing the matrix requires a destructive database change
- a migration would be required against live Supabase and cannot be safely staged
- existing endpoint consumers conflict with the proposed route separation
- required source data for default live metrics is unavailable across all supported snapshots
- verification exposes a security/data-integrity risk
- completing the matrix milestone appears to require fully implementing task 53.0 or 54.0

If completing the matrix appears to depend on full 53.0 or 54.0 implementation, document the dependency clearly and keep the matrix implementation focused on available metrics and planned placeholders.

Otherwise, make the best safe implementation decisions from the existing repo patterns and task notes, and continue until the acceptance criteria are met.
